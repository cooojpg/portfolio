AWS JAM 部内開催 要件定義書
1人運営・2〜3時間完結・Terraform一括デプロイ版

作成日: 2026年5月6日
形式: txt
対象: 部内向け小規模AWS JAM風ハンズオンイベント


1. 作りたいものを確認する

本企画で作りたいものは、部内向けの小規模 AWS JAM 風ハンズオンイベントである。

参加者は3チームに分かれ、Terraformで事前に一括デプロイされたAWS課題環境を操作し、制限時間内に課題を解いて得点を競う。

イベントでは、通常課題に加えて、早く解き終わったチーム向けのチャレンジ問題とエクストラ課題を用意する。

作成するものは以下とする。

- チーム別参加者ページ
- 管理者ページ
- スコアボード
- 自動採点API
- 管理者ページからの手動減点機能
- 通常課題
- チャレンジ問題
- エクストラ課題
- Terraformによる課題環境・採点基盤の一括デプロイ／一括削除

初回開催では、複雑な認証や高度な運営機能は作り込まず、1人で準備・運営・後片付けできる範囲に絞る。


2. 背景・目的を確認する

背景として、部内メンバーにAWSを実際に触る機会を提供したい。

座学だけではなく、実際のAWS環境を操作して課題を解くことで、AWSの基本的なサービス理解を深めることを目的とする。

主な目的は以下とする。

- AWS初心者がAWSマネジメントコンソールを触るきっかけを作る
- EC2、S3、Lambda、IAM、CloudWatchなどの基本サービスを体験する
- チームで相談しながら課題を解くことで部内コミュニケーションを促進する
- AWS認定資格学習への動機づけにする
- 障害対応風のチャレンジ問題を通じて、CloudWatch LogsやCloudWatch Metricsを使った調査の流れを体験する
- 1人運営でも再開催できるイベントテンプレートを作る


3. 利用者を確認する

本イベントの利用者は、参加者と管理者に分ける。

参加者:
- 部内メンバー
- AWS初心者〜CLF相当の知識を持つ人
- 3チーム構成
- 1チームあたり2〜3名程度を想定
- 各チームに専用URLを共有する

管理者:
- イベント運営者1名
- 事前準備、当日進行、減点登録、スコア確認、結果発表、後片付けを担当する
- 管理者専用URLから管理者ページにアクセスする


4. 主要機能を確認する

主要機能は以下とする。

4.1 チーム別参加者ページ

チームごとに異なる推測困難なURLを発行する。

例:
- /team-a-7f3k2/
- /team-b-9xq1m/
- /team-c-4p8z6/

参加者ページでは、以下を可能とする。

- 課題一覧の確認
- 課題詳細の確認
- 採点実行
- 自チームの得点確認
- 全体順位の確認
- 通常課題完了後のチャレンジ問題確認
- チャレンジ問題完了後のエクストラ課題確認

初回開催では、実装を簡略化するため、チャレンジ問題やエクストラ課題を最初から表示してもよい。


4.2 自動採点機能

参加者が採点実行ボタンを押すと、採点Lambdaが対象チームのAWS環境を確認する。

採点Lambdaは、課題ごとに定義された合格条件を満たしている場合に自動で加点する。

同一チーム・同一課題への加点は1回のみとし、複数回採点しても重複加点されないものとする。


4.3 手動減点機能

ヒント使用、生成AI使用、ルール違反、運営判断による調整は、管理者ページから手動で減点登録する。

最終スコアは以下で算出する。

最終スコア = 自動加点合計 - 手動減点合計 + 手動ボーナス点


4.4 管理者ページ

管理者ページは、推測困難な管理者専用URLで提供する。

例:
- /admin-8f3a2c9e/

管理者ページでは、以下を可能とする。

- 全チームのスコア確認
- 課題別の自動加点状況確認
- 手動減点登録
- 手動ボーナス登録
- 減点履歴確認
- 最終スコア確認
- 順位確認


4.5 スコアボード

スコアボードでは、以下を表示する。

- チーム名
- 自動加点合計
- 手動減点合計
- 手動ボーナス点
- 最終スコア
- 解答済み課題数
- 最終正解時刻
- 順位


4.6 課題構成

通常課題:
1. S3静的サイトを公開せよ
2. EC2 Webサーバーにアクセスせよ
3. Lambdaのエラーを修正せよ
4. IAM最小権限を設定せよ

チャレンジ問題:
5. インシデント対応：昨日まで使えていた社内Webページが今朝から開きにくい

エクストラ課題:
6. 今回の障害を再発させないための改善案を3つ提出せよ


5. 管理者向け機能と利用者向け機能を確認する

5.1 利用者向け機能

利用者向け機能は以下とする。

- チーム専用URLから参加者ページにアクセスする
- 課題一覧を確認する
- 課題詳細を確認する
- 採点実行ボタンを押す
- 採点結果を確認する
- 自チームの得点を確認する
- 全体順位を確認する
- 通常課題を完了した後、チャレンジ問題に取り組む
- チャレンジ問題も完了した後、エクストラ課題に取り組む


5.2 管理者向け機能

管理者向け機能は以下とする。

- 管理者専用URLから管理者ページにアクセスする
- 全チームのスコアを確認する
- 課題別の自動加点状況を確認する
- チーム・課題・理由・点数を指定して手動減点を登録する
- エクストラ課題の内容を確認し、必要に応じて手動ボーナスを登録する
- 減点履歴を確認する
- 最終スコアを確認する
- 順位を確認する


5.3 手動減点理由

管理者ページで選択できる減点理由は以下とする。

- ヒント使用
- 生成AI使用
- ルール違反
- 運営調整

減点例:
- ヒント使用: -10点
- 生成AI使用: -20点
- ルール違反: 任意
- 運営調整: 任意

減点点数は、理由ごとに初期値を設定するが、管理者が必要に応じて変更できるものとする。

7. 外部連携を確認する

初回開催では、外部SaaS連携は必須としない。

使用する外部・周辺ツールは以下に限定する。

- Terraform
- Git
- ブラウザ
- TeamsまたはZoom

スコア管理、採点、減点、スコアボード表示はAWS上で完結させる。

8. 認証・認可を確認する

8.1 認証方針

初回開催では、本格的なログイン機能は実装しない。

参加者ページは、チームごとに異なる推測困難なURLを発行し、そのURLを知っている参加者が該当チームとして利用できる方式とする。

管理者ページも、推測困難な専用URLを知っている人だけがアクセスできる方式とする。


8.2 チーム識別

参加者のチーム識別は、チーム別URLおよびチームトークンにより行う。

参加者ページは、URLパスに応じたチームトークンを保持し、採点実行時にAPIへ送信する。

採点Lambdaは、DynamoDB上のteamsテーブルでチームトークンを照合し、該当チームのリソースを採点する。


8.3 認可方針

参加者ページで可能な操作:
- 課題確認
- 採点実行
- スコア確認

管理者ページで可能な操作:
- 全体スコア確認
- 手動減点登録
- 手動ボーナス登録
- 減点履歴確認

管理者ページのURLは参加者に共有しない。

本方式は厳密な認証ではなく、部内小規模イベント向けの簡易アクセス制御とする。


8.4 将来的な改善

第2回以降、参加人数が増える場合や外部参加者を含む場合は、Cognito等によるログイン認証を追加する。


9. 非機能要件を確認する

9.1 開催規模

- 3チーム
- 1チーム2〜3名程度
- 運営者1名


9.2 所要時間

- 全体で2〜3時間
- 競技時間は90〜120分程度
- 通常課題で60〜90分程度
- チャレンジ問題で20〜30分程度
- エクストラ課題は余った時間で実施


9.3 可用性

初回開催では高可用構成は必須としない。

ただし、当日前に以下を確認する。

- terraform applyが成功すること
- 参加者ページにアクセスできること
- 管理者ページにアクセスできること
- 自動採点が動くこと
- 手動減点が登録できること
- terraform destroyが成功すること


9.4 コスト

小規模構成とし、不要に高額なAWSサービスは使わない。

初回開催で避けるもの:
- NAT Gateway
- RDS
- EKS
- SageMaker
- 高額なEC2インスタンス
- 長時間稼働する不要リソース

イベント終了後はterraform destroyで削除する。


9.5 セキュリティ

- 本番環境や顧客データとは分離する
- 個人情報や機密情報を課題データに含めない
- 管理者ページにAWS認証情報を表示しない
- URLは開催ごとに変更する
- チームURLを他チームへ共有しない前提とする
- 管理者URLは運営者のみが保持する


9.6 運用性

- 1人でデプロイ、進行、採点確認、減点登録、削除ができること
- 管理者ページは最小機能に絞る
- 減点登録時には確認ダイアログを表示する
- 課題数を増やしすぎない


9.7 再現性

- Terraformで同じ環境を再作成できること
- 第2回以降も再利用できる構成にする
- リソース名にはイベント名、チームID、課題IDを含める


10. 技術方針と運用方針を確認する

10.1 技術方針

フロントエンド:
- S3 + CloudFrontで静的サイトとして配信する
- 参加者ページと管理者ページを同一フロントエンド内に配置する
- チーム別URLと管理者URLはランダム文字列を含むパスにする

API:
- API Gateway + Lambdaで実装する
- 採点API、スコア取得API、減点登録API、ボーナス登録APIを用意する

データストア:
- DynamoDBを利用する
- チーム、課題、スコア、減点履歴、ボーナス履歴、提出履歴を保存する

採点:
- 採点LambdaがAWS SDKを使って対象リソースを確認する
- 課題ごとに合格条件を定義する
- 合格時にDynamoDBへスコアを記録する
- 同一課題の重複加点を防止する

Terraform:
- 課題環境とスコアボード基盤をTerraformで一括デプロイする
- イベント終了後はterraform destroyで一括削除する
- リソース名にはチームIDを含める


10.2 通常課題の技術方針

通常課題は、AWS初心者が基本サービスを触れる内容にする。

通常課題案:
1. S3静的サイトを公開せよ
   - S3バケット、index.html、静的Webサイトホスティング、公開設定を扱う
   - 採点はHTTP 200とFLAG表示で判定する

2. EC2 Webサーバーにアクセスせよ
   - EC2、Security Group、HTTPアクセスを扱う
   - 採点はHTTP 200とFLAG表示で判定する

3. Lambdaのエラーを修正せよ
   - Lambda、環境変数、CloudWatch Logsを扱う
   - 採点はLambda実行結果またはAPIレスポンスで判定する

4. IAM最小権限を設定せよ
   - IAMポリシー、S3権限、最小権限を扱う
   - 採点は対象バケットへの許可と対象外バケットへの拒否で判定する


10.3 チャレンジ問題の技術方針

チャレンジ問題は、通常課題をすべて完了したチーム向けの任意課題とする。

課題名:
インシデント対応：昨日まで使えていた社内Webページが今朝から開きにくい

シナリオ:
社内向けの簡易Webページが、昨日までは問題なく利用できていた。
しかし本日朝から、利用者より以下の問い合わせが入っている。

- ページがなかなか開かない
- たまに500エラーになる
- 何回か更新すると表示されることもある
- 昨日までは普通に見えていた

完全に停止しているわけではないが、通常より明らかに不安定な状態である。

参加者は運用担当として、Webページへアクセスして事象を確認し、CloudWatch LogsやCloudWatch Metricsを使って原因を特定する。
原因を特定したうえでEC2上の状態を修正し、Webページが安定して表示される状態へ復旧する。

想定構成:
- EC2
- Security Group
- SSM接続用IAM Role
- 簡易Webアプリ
- CloudWatch Logs
- 必要に応じてCloudWatch Agent
- 採点Lambda

障害原因:
- アプリケーションログが肥大化している
- EC2のルートボリューム空き容量が不足している
- Webアプリが一時ファイルまたはログを書き込めず、500エラーを返している

CloudWatchで確認できる情報:
- CloudWatch Logsに「No space left on device」などのエラーが出ている
- CloudWatch MetricsではCPUUtilizationは大きく異常ではない
- CloudWatch Agentを入れる場合はdisk_used_percentが高い

参加者が行うこと:
1. Webページへアクセスし、事象を確認する
2. ページが遅い、または500エラーになることを確認する
3. CloudWatch Logsでアプリケーションログを確認する
4. No space left on device などのエラーを確認する
5. EC2へSession Manager等で接続する
6. df -h でディスク使用率を確認する
7. 課題用に用意された不要ログファイルを削除する
8. 必要に応じてWebサーバーまたはアプリケーションを再起動する
9. Webページが安定して表示されることを確認する
10. 採点実行ボタンを押す

ゴール:
Webページが安定して表示され、ページ内に以下のFLAGが表示されること。

FLAG{disk-cleanup-restored}

制約:
- EC2インスタンスを新規作成し直さない
- アプリケーションを別サービスへ移行しない
- 既存のWebアプリを復旧する
- 原因調査にはCloudWatch LogsまたはCloudWatch Metricsを利用する
- 削除対象は課題用に用意された不要ログに限定する
- 本番データや個人情報を扱わない

ヒント:
ヒント1:
Webページのエラーだけでなく、アプリケーションログを確認してください。

ヒント2:
CloudWatch Logsに No space left on device というエラーが出ていないか確認してください。

ヒント3:
EC2に接続し、df -h でディスク使用率を確認してください。

採点条件:
- 対象URLにHTTP 200でアクセスできる
- レスポンス本文に FLAG{disk-cleanup-restored} が含まれる
- 同一チーム・同一課題への加点は1回のみとする

より厳密にする場合の追加採点条件:
- 採点Lambdaが対象URLへ複数回アクセスし、一定回数以上HTTP 200が返ること
- EC2上のディスク使用率が一定値未満であること
ただし、初回開催では簡易採点でよい。


10.4 エクストラ課題の技術方針

エクストラ課題は、チャレンジ問題まで完了したチーム向けの任意課題とする。

課題名:
今回の障害を再発させないための改善案を3つ提出せよ

内容:
今回のWebページ不安定化インシデントを踏まえ、再発防止策を3つ提案する。

改善案の例:
- ログローテーションを設定する
- ディスク使用率のCloudWatch Alarmを作成する
- CloudWatch Agentでdisk_used_percentを監視する
- アプリケーションログの出力量を見直す
- 定期的な不要ログ削除を仕組み化する
- 障害対応Runbookを作成する
- Terraformで監視設定を標準化する
- 500エラー率を監視する
- 定期的な運用チェック項目にディスク容量確認を入れる

採点:
- 自動採点の対象外とする
- 管理者ページから手動でボーナス点を登録する
- 最大50点、または特別賞扱いとする


10.5 運用方針

事前:
- Terraform applyを実行する
- 各チームURLと管理者URLを確認する
- 自動採点が動くことを確認する
- 手動減点が登録できることを確認する
- チャレンジ問題のWebページが意図した不安定状態になることを確認する
- terraform destroyの手順を確認する

当日:
- チームごとに専用URLを共有する
- 参加者は各自のチームURLから参加する
- 管理者は管理者ページでスコアと減点を管理する
- 減点はヒント使用、生成AI使用、ルール違反、運営調整を対象とする
- チャレンジ問題は通常課題を完了したチーム向けに案内する
- エクストラ課題はチャレンジ問題完了後の待ち時間対策として案内する

終了後:
- 最終スコアを確認する
- 表彰する
- terraform destroyを実行する
- 残存リソースを確認する


11. やらないことを確認する

初回開催では、以下はやらない。

- 本格的なログイン認証
- Cognito認証
- 社内IDP連携
- 複雑な管理者権限管理
- Slack通知連携
- Googleスプレッドシート連携
- Notion連携
- 参加者ごとの個人アカウント管理
- 高度な監査ログ機能
- マルチアカウント前提の複雑な構成
- EKS、ECS、RDS、NAT Gatewayなど課金や準備が重い課題
- プロジェクターや専用機材を前提にした運営
- 8問以上の大量課題
- 運営者が複数人いないと成立しない設計
- 本番データや顧客データを使う課題
- 参加者に危険な削除操作を自由に行わせる課題
- 高度な負荷試験を伴う課題


12. 完了条件を確認する

12.1 イベント準備の完了条件

- Terraformで課題環境と採点基盤を一括デプロイできる
- 3チーム分の参加者URLが発行されている
- 管理者URLが発行されている
- 参加者ページから採点実行できる
- 管理者ページから減点登録できる
- 管理者ページからボーナス登録できる
- スコアボードに自動加点、手動減点、手動ボーナス、最終スコアが表示される
- 通常課題、チャレンジ問題、エクストラ課題が整理されている
- チャレンジ問題で意図した障害状態が再現できている
- CloudWatch Logsで原因調査に必要なログが確認できる
- terraform destroyで一括削除できる


12.2 イベント当日の成功条件

- 全チームが参加者ページにアクセスできる
- 全チームが1問以上クリアできる
- 自動採点が正常に動作する
- 管理者が手動減点を登録できる
- 最終順位を確認できる
- チャレンジ問題に少なくとも1チームが着手できる
- イベント後にAWSリソースを削除できる


12.3 学習面の成功条件

- 参加者がAWS基本サービスを実際に触れる
- チームで相談しながら課題を解ける
- CloudWatch LogsやCloudWatch Metricsを使った基本的な原因調査を体験できる
- AWSを業務や資格学習で使うイメージを持てる
