General Design Guide

Agent Team はどう設計すべきか

特定プロジェクトの設定に依存せず、Claude / ChatGPT / Google系の公式情報や実務上の失敗パターンを踏まえて、 マルチAgentチームを設計するときの原則を整理したガイドです。

1. 基本結論

Agent Team は「常に使うもの」ではなく、「分解・並列調査・検証が必要なときに使うもの」です。
単純作業は単体Agentで十分です。複雑な調査、複数領域の設計、レビューが必要な実装、セキュリティ・運用影響がある変更ではTeam化が有効です。
使うべき場面 リサーチ、設計、複数領域変更、レビュー、検証、移行計画。
使いすぎ注意 小さい修正、順序依存が強い作業、scopeが曖昧な作業。
推奨構成 中央Coordinator + 必要なSpecialist + Worker + Reviewer。

2. 推奨アーキテクチャ

最も扱いやすいのは、中央のCoordinatorが全体を見て、必要なAgentにだけ委譲する hub-and-spoke型 です。

役割 責務 注意点
Router タスクを分類し、Agent Teamを使うか決める。 ここで無駄な起動を止める。
Coordinator タスク分解、Agentへの委譲、結果の統合、最終判断。 自分で全部実装しない。判断と統合に集中する。
Planner 方針、作業順序、未決事項、リスクを整理する。 順序依存がある作業では先に起動する。
Specialist 特定領域の調査・観点出し・制約確認。 原則read-onlyにすると衝突しにくい。
Worker / Implementer 限定された範囲で実装・修正・ドキュメント作成を行う。 編集対象を明確にする。同一ファイルの並列編集は禁止。
Reviewer 欠陥、要件漏れ、過剰変更、一貫性を確認する。 実装後だけでなく、必要なら作業前の観点出しにも使う。
Tester 検証手順、テスト観点、回帰リスクを確認する。 実行可能な検証コマンドや期待結果まで出す。
Security / DevOps 権限、公開境界、秘密情報、CI/CD、運用影響を確認する。 常時起動ではなく、関係する変更時だけでよい。

3. 標準フロー

分類
計画
分担
作業
レビュー
修正
  1. 分類: タスクを S / M / L / X に分け、Team化が必要か判断する。
  2. 計画: 仕様、リスク、順序依存、編集範囲、検証観点を決める。
  3. 分担: Agentごとに目的、入力、対象範囲、禁止事項、出力形式を明確に渡す。
  4. 作業: Workerは限定されたscopeだけを編集する。
  5. レビュー: Reviewer / Tester / Security / DevOpsが欠陥や不足を確認する。
  6. 修正: blocking findingが残る場合は、完了せず最小scopeで修正して再レビューする。

4. タスク複雑度と推奨Agent数

複雑度 目安 推奨Agent数 構成例
S 単一ファイル、typo、小さい修正 0〜1体 本体処理、またはImplementerのみ
M 2〜5ファイル、1〜2領域、既存パターンへの追加 2〜4体 Planner + Specialist + Implementer + Reviewer
L 新機能、複数領域、API変更、設計変更 4〜7体 Coordinator + Planner + Specialist群 + Implementer + Reviewer + Tester
X 認証境界、データ形式、移行、広範囲リファクタ 6〜9体 Coordinator主導 + Security + DevOps + Tester + 複数Specialist
例外 大規模な並列調査・横断レビュー 10〜12体 明確に分割できる調査単位がある場合のみ
目安: 通常運用は 3〜7体が扱いやすいです。10体以上は「明確に並列化できる」「統合役がいる」「出力形式が揃っている」場合だけにします。

5. Agentに渡す指示テンプレート

Agent Teamで一番重要なのは、各Agentへの指示を曖昧にしないことです。

## Mission
あなたの役割は {role} です。
今回の目的は {goal} です。

## Scope
対象範囲:
- {paths_or_domains}

対象外:
- {out_of_scope}

## Input
参照すべき情報:
- {requirements}
- {existing_files}
- {constraints}

## Output
返す形式:
- 結論
- 根拠
- 変更が必要な箇所
- リスク
- 未決事項
- 次に起動すべきAgentがあれば理由つきで提案

## Rules
- scope外の作業をしない
- 不明点は推測せず「未決事項」に入れる
- 同一ファイルを他Workerと並列編集しない
- blocking finding は明確にラベル付けする

6. 並列化してよいもの / だめなもの

並列化してよい
  • 独立したリサーチ
  • 別領域のread-onlyレビュー
  • 別ファイル・別責務の実装
  • セキュリティ観点とテスト観点の並行確認
  • 複数案の生成と比較
並列化しない方がよい
  • 同一ファイルの編集
  • API仕様が未確定なfrontend / backend同時実装
  • データモデル変更と利用側実装の同時進行
  • 認証・権限境界の判断が未確定な作業
  • 最終文体や設計一貫性が重要な成果物

7. Teamを使うべきではないケース

ケース 理由 代替
typoや単一ファイルの軽微修正 調整コストのほうが大きい。 単体Agentで即修正。
順序依存が強い作業 前工程の決定がないと後工程がブレる。 まずPlannerで仕様確定。
scopeを明確に切れない作業 Agent間で重複・漏れ・過剰作業が起きる。 人間またはCoordinatorが先に分解。
検証方法がない作業 レビューが主観になりやすい。 先に期待結果と検証観点を作る。
「念のため」だけで増やす コスト、遅延、文脈汚染が増える。 必要な観点だけチェックリスト化。

8. よくある設計パターン

パターン 向いている用途 構成
Research Team 網羅調査、競合調査、技術比較 Coordinator + 複数Research Specialist + Citation / Reviewer
Implementation Team 中規模実装、既存機能追加 Planner + Domain Specialist + Implementer + Reviewer + Tester
Architecture Review Team 設計変更、クラウド構成、認証境界 Architect / Security / DevOps / Cost / Reviewer
Migration Team データ形式変更、API移行、互換性対応 Coordinator + Data Specialist + Implementer + Tester + Rollback Planner
Content Team SEO記事、ドキュメント、提案書 Researcher + Structure Writer + Writer + Editor + Fact Checker

9. 最小ルールセット

まずはこのルールだけでも入れると、Agent Teamはかなり安定します。

## Agent Team 基本ルール

1. Agent Teamは、タスクが分解可能で、並列調査またはレビュー価値がある場合のみ使う。
2. 小さい修正は単体Agentで処理する。
3. 中央Coordinatorが最終判断と統合を担当する。
4. 子Agentはさらに子Agentを起動しない。
5. Specialistは原則read-onlyにする。
6. Workerには編集対象、禁止事項、期待出力を明確に渡す。
7. 同一ファイルを複数Workerで並列編集しない。
8. Reviewer / Tester / Securityは必要なときだけ起動する。
9. blocking findingが残る場合は完了扱いにしない。
10. Agent数は通常3〜7体、多くても明確な理由がある場合のみ10体以上にする。

10. 公式情報からの設計根拠

  • Anthropic Engineering: How we built our multi-agent research system
    Lead agentがsubagentへ目的・出力形式・利用ツール・情報源・境界を明確に渡す必要がある、という設計根拠。
  • Anthropic Engineering: Effective context engineering for AI agents
    Contextは有限であり、Agentに渡す情報を設計する必要がある、という根拠。
  • OpenAI Agents SDK: Agent orchestration
    Agentの実行順序、判断方法、handoffをどう制御するかという設計根拠。
  • OpenAI API Docs: Agents SDK
    Agentsは計画し、ツールを呼び、specialistと協調し、multi-step workを行うアプリケーションという定義。
  • Google Research: Towards a science of scaling agent systems
    Multi-agentはタスク特性次第で改善も悪化もするため、増やしすぎず、タスクに応じて設計すべきという根拠。
